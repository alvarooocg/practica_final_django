Práctica Final: Plataforma de Gestión de Actividades para un Centro Cultural

Contexto de la aplicación

Un centro cultural municipal desea digitalizar la gestión de sus actividades y talleres para mejorar la experiencia de los usuarios y facilitar el trabajo del personal. Actualmente, todas las inscripciones y asignaciones se hacen en papel, lo que provoca errores frecuentes, duplicidades y pérdida de información.

Se te encarga desarrollar una aplicación web con Django que permita registrar las actividades, gestionar a los usuarios que se inscriben en ellas, y administrar a los monitores que las imparten. Además, cada actividad puede estar asociada a varias salas del edificio, que pueden compartirse entre actividades siempre que no coincidan en horario.

Tu tarea será diseñar la arquitectura del sistema, justificar tus decisiones mediante el modelo C4, razonar sobre patrones de diseño, y por supuesto, construir el sistema web funcional.

Parte práctica (obligatoria)

Dominio de la aplicación y requisitos

Tu sistema de gestión deberá contemplar las siguientes entidades y relaciones:

  • Actividad (nombre, tipo, horario, descripción, duración, plazas disponibles).

  • Usuario inscrito (nombre, edad, email, teléfono).

  • Monitor (nombre, especialización, número de actividades asignadas).

  • Sala (nombre, capacidad, ubicación).

Relaciones requeridas:

  • Un Usuario puede inscribirse a muchas Actividades y una Actividad puede tener muchos Usuarios inscritos (N a N).

  • Una Actividad tiene asignado un único Monitor, pero un Monitor puede impartir varias Actividades (1 a N).

  • Una Actividad puede usar una única Sala principal, pero también puede requerir otras Salas secundarias, compartidas con otras actividades si no hay solapamiento (N a N).

  • Cada Sala tiene un único responsable técnico (ResponsableSala) (1 a 1).

Funcionalidad mínima obligatoria

  • Definir los modelos y relaciones en Django.

  • Crear templates funcionales para gestionar la navegación por la información.

  • La navegación web y el estilo gráfico son libres, pero deben ser funcionales.

A continuación, se ofrece una lista de endpoints necesarios para cubrir toda la funcionalidad mínima de la aplicación:

Endpoints sugeridos

Actividades:

  • /actividades/ → listado general de actividades

  • /actividades/nueva/ → formulario para crear una nueva actividad

  • /actividades/<id>/ → detalle de una actividad específica

  • /actividades/<id>/editar/ → edición de actividad

  • /actividades/<id>/eliminar/ → eliminación de actividad

Usuarios inscritos:

  • /usuarios/ → listado general de usuarios inscritos

  • /usuarios/nuevo/ → formulario para dar de alta un usuario

  • /usuarios/<id>/ → detalle del usuario

  • /usuarios/<id>/editar/ → edición de usuario

  • /usuarios/<id>/eliminar/ → eliminación de usuario

Monitores:

  • /monitores/ → listado general de monitores

  • /monitores/nuevo/ → creación de nuevo monitor

  • /monitores/<id>/ → detalle de monitor

  • /monitores/<id>/editar/ → edición

  • /monitores/<id>/eliminar/ → eliminación

Salas:

  • /salas/ → listado de salas

  • /salas/nueva/ → creación de sala

  • /salas/<id>/ → detalle de la sala

  • /salas/<id>/editar/ → edición

  • /salas/<id>/eliminar/ → eliminación

Inscripciones:

  • /actividades/<id>/inscripciones/ → listar usuarios inscritos en una actividad

  • /actividades/<id>/inscribir/ → formulario de inscripción de usuario

  • /actividades/<id>/inscripciones/<usuario_id>/eliminar/ → cancelar inscripción

Filtros y búsquedas (pueden implementarse como vistas o con parámetros GET):

  • /actividades/?tipo=danza → filtrar por tipo de actividad

  • /actividades/?monitor=3 → filtrar por monitor

  • /usuarios/?actividad=5 → mostrar usuarios de la actividad 5
